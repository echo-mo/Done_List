# 新增需求：实现移动端“左滑删除”交互 (Swipe-to-Delete)

前置重构非常成功。现在我们需要提升移动端的用户体验：请将任务列表中的“删除”操作，改为移动端主流的“向左滑动任务，露出底部的 Delete 按钮，然后再点击删除”的设计。

为了不破坏现有的稳固架构，请按以下技术要求提供修改方案：

## 1. 结构与样式改造 (CSS 增强)
- 调整 `.task` (li) 容器，使其成为 `overflow: hidden` 和 `position: relative` 的视口。
- 将 `.btn-delete` 从原本的 `.actions` 流式布局中剥离，设置为绝对定位（`position: absolute; right: 0`），隐藏在任务内容的下层。
- 任务的核心可见内容（`.task-content` 及 Undo/Edit/Done 按钮）需要在一个可以横向位移（`transform: translateX`）的顶层容器中。
- 加入平滑的 CSS 回弹动画（`transition: transform 0.3s ease`）。

## 2. 触摸事件处理 (JS - ui.js 修改)
请在 `ui.js` 中新增对 `touchstart`, `touchmove`, `touchend` 的监听。为了性能，依然推荐使用事件代理。
- **滑动阈值**：左滑时，顶层容器跟随手指滑动（最大移动距离例如 80px）。松手时，若滑动超过阈值（如 40px），则吸附到展开状态；否则回弹关闭。
- **滚动防抖（防冲突核心）**：在 `touchmove` 初期，必须判断 X 轴和 Y 轴的位移比例。如果是垂直滑动（上下看列表），必须立刻放弃水平位移逻辑，绝不能阻止浏览器的默认滚动。
- **互斥与恢复体验**：
  - 全局同一时间只能有一个任务处于“左滑展开”状态。开始滑动新任务时，必须自动收起其他已展开的任务。
  - 用户点击页面的其他任何空白区域，自动收起当前展开的删除按钮。

## 3. 兼容性保护
- 请确保我们之前做的“手机端 Edit 状态保护（`Store.isEditing`）”逻辑不受影响。
- 桌面端如果不支持 Touch 事件，可以考虑通过鼠标拖拽（Mouse Events）兼容，或者直接保留 Hover 显式 Delete 按钮的降级方案。
